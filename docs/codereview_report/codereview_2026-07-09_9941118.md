## 安全性發現

- [x] **[LOW]** Owner token 比較非 constant-time　✅ 已修復（9941118 修復分支）
  - **位置**：`backend/app/api/auth.py:60`
  - **描述**：`x_ingest_owner_token != secret` 以一般字串比較，理論上可透過回應時間差做 timing attack 逐字元還原 secret。此端點已在 admin key 之後，且屬 portfolio demo，風險低。

- [x] **[LOW]** Owner token 以明文存於 localStorage　✅ 已修復（改用 sessionStorage）
  - **位置**：`frontend/app.js:562-564`（`OWNER_STORE` / `setOwner`）
  - **描述**：擁有者權杖持久化在 `localStorage`，任何 XSS 皆可讀出並外洩此長期密鑰。前端其餘部分未見 innerHTML 注入不可信內容（ingest 區塊皆以 textNode 插入），故 XSS 面偏小，但密鑰落地仍是殘餘風險。
  - **建議**：可接受的 demo 取捨；若要降風險，改用 `sessionStorage` 或不持久化，讓每次操作重新輸入。至少在 README 註明此權杖會存在瀏覽器。
  - **修復**：`getOwner`/`setOwner` 改用 `sessionStorage`，權杖只存活於當前分頁 session、不再跨造訪落地。UX 取捨：關閉分頁後需重新輸入。

## 邏輯正確性發現

- [x] **[MEDIUM]** run 失敗時的跨儲存體部分寫入，無交易保護　✅ 已修復（縮小失敗視窗）
  - **位置**：`ingestion/extract/runner.py:255-262`
  - **描述**：run 流程依序 `upsert_documents → delete_chunks_for_doc(PG) → delete_chunks_for_doc(Qdrant) → upsert_chunks(PG) → embed_chunks() → load_qdrant()`。asyncpg 為 autocommit，且三個儲存體（PG chunks、Qdrant vectors、curation_items）之間無交易。若 `embed_chunks()`（可能呼叫 OpenAI embeddings）或 `load_qdrant.load_chunks()`（Qdrant 不可用）在「Qdrant 已刪除舊點」之後拋錯，會留下 PG 有新 chunk、Qdrant 卻空的狀態，導致檢索時 chunk metadata 存在但無向量。此外每個 chunk 的 `stage_extraction_output` 已即時 commit 進 `curation_items`，即使整個 job 最終標記 failed 也不會回滾。屬 owner-locked、可重跑的 admin 動作，衝擊有限但仍為真實的一致性缺口。
  - **建議**：最小改動——把 `embeddings = embed_chunks(chunk_rows)` 移到任何 delete 之前計算（embedding 只依賴 chunk 內容，不依賴 DB），使最易失敗的外部步驟先完成，再連續執行 delete/upsert(PG)/load(Qdrant)，縮小失敗視窗。進一步可將 PG 寫入包進單一 asyncpg transaction，並把 Qdrant 的 delete+load 併到 PG commit 成功之後。
  - **修復**：已將 `embed_chunks` 提前到所有 delete 之前計算，最易失敗的 embedding 呼叫先完成，delete/upsert/load 才連續執行。PG transaction 包裹為選配後續，本次未做（低頻 owner-locked 且可重跑，殘餘視窗已顯著縮小）。

- [x] **[LOW]** proposed 統計在 `ON CONFLICT DO NOTHING` 下會高估　✅ 已修復（RETURNING 實際入列數）
  - **位置**：`ingestion/extract/runner.py:244-250`；配合 `ingestion/pipeline/load_postgres.py:95,104`
  - **描述**：`stage_extraction_output` 以 `ON CONFLICT (item_id) DO NOTHING` 寫入 curation_items，但 runner 無論是否真的插入，都以 `proposed_nodes += len(node_ids)` 計數並把 id 回報為該 chunk 的 `proposed_node_ids`。當兩個 chunk 提出相同 concept id、或重跑同一 doc 時，實際被跳過的項目仍被算進統計與前端顯示，數字會高於真正入列的候選數。
  - **建議**：讓 `stage_extraction_output` 回傳實際插入的 id 清單（`INSERT ... ON CONFLICT DO NOTHING RETURNING item_id`），runner 依實際回傳計數；或在文件/UI 註明此為「提出數」而非「入列數」。
  - **修復**：`stage_extraction_output` 改用 `fetchrow ... RETURNING item_id` 回傳 `(ok, error, staged_nodes, staged_edges)`，runner 的 `proposed_nodes/edges` 改累加實際入列數。per-chunk 的 `proposed_node_ids` 維持為該 chunk 抽取到的完整 id（作為 concept 參照仍正確）。兩處測試解包與 `test_extraction_validation.py` 已同步更新。

- [ ] **[LOW]** 重跑 doc 只清 chunks，不清前次殘留的 proposed curation_items　⏸ 刻意保留
  - **位置**：`ingestion/extract/runner.py:258`
  - **描述**：重新 ingest 時 `delete_chunks_for_doc` 清掉 PG/Qdrant 舊 chunk，但前一次 run 已 staged 的 `curation_items`（proposed 節點/關係）不會被清理。若換了抽取策略導致 id 改變，舊的 proposed 項會滯留在審訂佇列。因 curation 為人工審核，非致命，但會累積雜訊。
  - **建議**：可接受（人工審核會處理）；若要更乾淨，重跑前依 doc 來源刪除該 doc 尚未審核的 proposed 項，或在 curation UI 標示來源 job_id 以利辨識。
  - **未修復（刻意）**：`curation_items` 目前無 doc/job 關聯欄位，要「依 doc 清除未審核 proposed 項」需加 schema 欄位 + migration，對一個人工審核下屬美觀性的 LOW 而言不成比例（over-engineering）。列為後續選配，本次不動。

## 效能發現

- [x] **[MEDIUM]** 同步阻塞 I/O 在 async 端點內執行，阻塞事件迴圈　✅ 已修復（asyncio.to_thread）
  - **位置**：`ingestion/extract/runner.py:122`（`extract_fn`）、`261`（`embed_chunks`）、`262`（`load_qdrant`）；`ingestion/extract/llm_client.py:43`（同步 OpenAI client）
  - **描述**：`ingest_run` 為 async 端點，但 LLM 抽取、embedding、Qdrant 寫入皆為同步阻塞呼叫，且未以 `run_in_executor`/`asyncio.to_thread` 卸載。整個 ingest（每 chunk 一次 LLM 呼叫，可能數秒 × 多 chunk）期間會佔住事件迴圈，阻塞同進程其他所有請求。屬單一擁有者、低頻的 admin 操作，衝擊可控，但在展示時若同時有人瀏覽會明顯卡頓。
  - **建議**：把阻塞呼叫包進 `await asyncio.to_thread(...)`（stdlib），或改用 OpenAI/Qdrant 的 async client。最小改動為 `to_thread` 包裹 `extract_fn` 與 `load_qdrant.load_chunks`。
  - **修復**：`extract_fn`、`embed_chunks`、`load_qdrant.delete_chunks_for_doc`、`load_qdrant.load_chunks` 皆改以 `await asyncio.to_thread(...)` 卸載到 worker thread，ingest 期間不再佔住事件迴圈。同步的 fake_extract 測試在 thread 中仍正常運作。

- [x] **[LOW]** `build_user_prompt` 每個 chunk 重讀並 regex 解析 base prompt 檔　✅ 已修復（lru_cache）
  - **位置**：`ingestion/pipeline/build_extraction_prompt.py:63`（`_load_base_blocks`），經 `runner.py:219` 迴圈每 chunk 呼叫
  - **描述**：`build_user_prompt` 每次都 `read_text` + `_FENCE_RE.findall` 整份 base prompt 檔；N 個 chunk 即 N 次讀檔與 regex。檔案小，衝擊輕微，但屬可輕易消除的重複工。
  - **建議**：在 runner 迴圈外先取一次 user template（例如 `build_system_prompt` 已在迴圈外組好，可同樣把 user template 抽到迴圈外），或對 `_load_base_blocks` 加 `functools.lru_cache`。
  - **修復**：`_load_base_blocks` 加上 `@lru_cache(maxsize=1)`，讀檔與 regex 只做一次；smoke 驗證快取確實命中。

- [ ] **[LOW]** run 端點每次請求新建 asyncpg 連線，無連線池　⏸ 刻意保留
  - **位置**：`backend/app/api/routes_ingest.py:160`
  - **描述**：每次 run 以 `asyncpg.connect(...)` 建立單一連線，用完關閉，無 pool。低頻 admin 動作可接受。
  - **建議**：維持現狀即可；若未來 ingest 走高頻或並發，改用共用 pool。
  - **未修復（刻意）**：報告本身即建議維持現狀，低頻 owner-locked 動作不需連線池，加 pool 反而是多餘複雜度。
